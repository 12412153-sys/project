## 1. 整体架构与数据流

```
板载输入                    drink_vending_top                        板载输出
────────                  ┌────────────────────────────────┐        ────────
100MHz P17 ──────────────▶│ clk                            │
                          │    clk_wiz_0 → pclk(25.175MHz) │
btn[5:0] ──消抖×6────────▶│ btn_d[5:0]                     │
sw[7:0] ────────────────▶│                                 │
ps2_clk/data ──键盘────▶│ kbd_data[3:0], kbd_valid        │
                          │                                 │──▶ seg0/seg1/an  数码管
                          │  ┌─── sys_state ─────────────┐ │──▶ led[15:0]
                          │  │  MAIN_MENU / SALE / ADMIN  │ │──▶ aud_pwm/sd  蜂鸣
                          │  └────┬────────────┬──────────┘ │──▶ vga_r/g/b/hs/vs
                          │       │            │             │
                          │  sales_v3      admin_mode        │
                          │       │            │             │
                          │       └────────────┘             │
                          │         register_file            │
                          └────────────────────────────────┘
```

**核心设计思路**：
- `register_file` 是唯一数据源，所有显示模块读同一份数据 → 显示必然一致
- 顶层 `sys_state` 是"交通总指挥"，决定数码管/LED/VGA 显示哪个模块的数据
- 按键/键盘信号**共享**传给 `sales_v3` 和 `admin_mode`，由各模块的 `sales_en`/`admin_en` 使能信号决定谁响应

## 2. drink_vending_top.v — 顶层模块

### 2.1 功能概述

连接所有子模块，实现顶层三状态 FSM，完成信号路由、显示复用和 LED 输出。

### 2.2 顶层 FSM

```verilog
localparam MAIN_MENU = 2'd0, SALE_MODE = 2'd1, ADMIN_MODE = 2'd2;
reg [1:0] sys_state;
```

```
上电 / S5(PROG_B 重新配置)
        │
        ▼
   MAIN_MENU  →  数码管: "r1-0S-1A"  LED: 半亮（左=管理/右=销售）
   SW[0]=0, S2 ─────────────────────────────▶ SALE_MODE
   SW[0]=1, S2 ─────────────────────────────▶ ADMIN_MODE
        ▲                                           │
        └────────────────── S6(btn[5]) ─────────────┘
                           软退出，数据保留
```

**S6 的实现**：P15 引脚在板上有 4.7kΩ 上拉到 VCC，未按时为高电平，按下接地为低电平。顶层对 `btn[5]` 取反后再送消抖：
```verilog
btn_debounce u_deb5 (.btn_in(~btn[5]), .btn_out(btn_d[5]));
```
这样 `btn_d[5]=1` 就代表"S6 被按下"，与其他按键行为一致。

**为什么没有 rst_n 端口？**  
`rst_n` 在顶层内部固定为 `wire rst_n = 1'b1`，始终有效。真正的"全局复位"是按 S5（PROG_B）重新加载比特流——这等价于重新上电，`initial` 块中的初始值重新写入所有寄存器。S6 只是让 FSM 回主菜单，完全不影响寄存器数据。

### 2.3 FAIL_HOLD 参数

```verilog
module drink_vending_top #(
    parameter [29:0] FAIL_HOLD = 30'd500_000_000  // 5 s @ 100 MHz
)
```

这个参数向下传递给 `admin_mode`，控制密码单次输错后显示 `FAIL` 的时长。仿真时覆盖为小值（如 200 周期），避免仿真跑太久。

### 2.4 MMCM 像素时钟

```verilog
wire pclk;
wire mmcm_locked;
clk_wiz_0 u_clk_wiz (
    .clk_in1 (clk),       // 100 MHz 输入
    .reset   (1'b0),
    .locked  (mmcm_locked),
    .clk_out1(pclk)        // 25.175 MHz 输出给 VGA
);
```

`clk_wiz_0` 是 Vivado Clocking Wizard IP 生成的模块，内部使用 MMCM 硬件精确分频。**这个文件不是手写的**，需要在 Vivado IP Catalog 中生成。`mmcm_locked` 指示 MMCM 已稳定锁定，本设计未使用它（VGA 从上电就开始工作）。

### 2.5 数码管数据复用

```verilog
localparam [39:0] MENU_DISP = {5'd22,5'd1,5'd24,5'd0,5'd20,5'd24,5'd1,5'd10};
//                              r     1    -    0    S    -    1    A

always @(*) begin
    case (sys_state)
        MAIN_MENU:  disp_data = MENU_DISP;        // "r1-0S-1A"
        SALE_MODE:  disp_data = sale_view_data;   // 来自 sales_v3
        ADMIN_MODE: disp_data = admin_view_data;  // 来自 admin_mode
    endcase
end
```

### 2.6 LED 复用

| `sys_state` | LED 输出 | 含义 |
|---|---|---|
| MAIN_MENU | sw[0]=0 → `0x00FF`；sw[0]=1 → `0xFF00` | 右亮=销售，左亮=管理 |
| SALE_MODE | `sale_led_out`（来自 sales_v3） | 跟随销售状态变化 |
| ADMIN_MODE | 报警时 `0xAAAA`，否则 `0x0000` | 报警闪烁 |

### 2.7 admin_mode 的 drink 数据多路选择

`admin_mode` 一次只查看一种饮料，所以顶层根据管理员当前选中的 `admin_drink_id` 多路选择对应的 price/stock：

```verilog
always @(*) begin
    case (admin_drink_id_raw[1:0])
        2'd0: begin admin_cur_stock = rf_stock0; admin_cur_price = rf_price0; end
        2'd1: begin admin_cur_stock = rf_stock1; admin_cur_price = rf_price1; end
        ...
    endcase
end
```

而 `sales_v3` 接收全部 4 路 price/stock，自己内部根据 `drink_id` 选。

### 2.8 端口一览

| 端口 | 方向 | 说明 |
|------|------|------|
| `clk` | 输入 | 100 MHz 系统时钟（P17） |
| `btn[5:0]` | 输入 | S0–S4（高有效）+ S6/P15（低有效，取反后消抖） |
| `sw[7:0]` | 输入 | 拨码开关（仅 sw[0]/sw[1] 使用） |
| `ps2_clk/data` | 输入 | PS/2 键盘接口 |
| `seg0/seg1/an` | 输出 | 8 位数码管段线和位选（高有效） |
| `led[15:0]` | 输出 | LED 状态指示 |
| `aud_pwm/sd` | 输出 | 蜂鸣器 PWM 和使能 |
| `vga_r/g/b/hs/vs` | 输出 | VGA 信号 |

## 3. btn_debounce.v — 按键消抖

### 3.1 为什么需要消抖？

物理按键按下时，由于弹片机械抖动，电信号会在 1–20 ms 内多次跳变（0→1→0→1→0→1...）。如果直接用，会被系统误认为连续多次按下。

### 3.2 消抖原理

采用**计数器稳定法**：只有输入信号**持续高电平超过 CNT_MAX 个时钟周期**（20 ms），才输出一个单周期高脉冲。

```verilog
module btn_debounce #(
    parameter CNT_MAX   = 2_000_000,  // 50 Hz，20 ms debounce window @ 100 MHz
    parameter CNT_WIDTH = 25
)(
    input  clk, rst_n, btn_in,   // btn_in: active-high
    output reg btn_out            // one-cycle pulse, active-high
);

reg [CNT_WIDTH-1:0] cnt_20ms;

always @(posedge clk or negedge rst_n) begin
    if (!rst_n)
        cnt_20ms <= 0;
    else if (btn_in == 0)
        cnt_20ms <= 0;            // 按键松开或抖动低电平时立即清零
    else if (cnt_20ms == CNT_MAX-1)
        cnt_20ms <= cnt_20ms;     // 计满后保持，不溢出
    else
        cnt_20ms <= cnt_20ms + 1;
end

always @(posedge clk or negedge rst_n) begin
    if (!rst_n)
        btn_out <= 0;
    else if (cnt_20ms == CNT_MAX-2)  // 计满前一拍输出高脉冲
        btn_out <= 1;
    else
        btn_out <= 0;
end
```

**关键细节**：
- `btn_out` 只在 `cnt == CNT_MAX-2` 时为高，**持续仅 1 个时钟周期**（10 ns）。这保证了一次按键只触发一次动作，不论按住多久。
- 顶层为每个按键（共 6 个）各实例化一个 `btn_debounce`，参数完全相同。

## 4. ps2_keyboard.v — PS/2 键盘解码

### 4.1 PS/2 协议基础

PS/2 是一种串行协议，通过两根线（时钟 ps2_clk、数据 ps2_data）传输数据。每次传输 **11 位帧**：

```
起始位(0) | D0 | D1 | D2 | D3 | D4 | D5 | D6 | D7 | 奇校验位 | 停止位(1)
                  ← 数据位，LSB 先传 →
```

- 数据在 ps2_clk **下降沿**稳定，接收方在下降沿采样
- 每次按下一个键（make）发一帧扫描码，松开（break）发 `0xF0` + 扫描码两帧

### 4.2 关键实现细节

**三级同步器防亚稳态**：ps2_clk 是外部异步时钟，直接用会引起亚稳态。用三个 D 触发器同步后取 `clk_sr[2]&~clk_sr[1]` 检测下降沿：

```verilog
reg [2:0] clk_sr;
always @(posedge clk) clk_sr <= {clk_sr[1:0], ps2_clk};
wire ps2_fall = clk_sr[2] & ~clk_sr[1];  // 下降沿检测
```

**移位寄存接收**：

```verilog
if (ps2_fall) begin
    if (bit_cnt == 0) begin
        if (!ps2d) bit_cnt <= 1;          // 检测到起始位（低）
    end else if (bit_cnt <= 8) begin
        shift[bit_cnt-1] <= ps2d;         // D0..D7 依次填入
        bit_cnt <= bit_cnt + 1;
    end else if (bit_cnt == 9) begin
        bit_cnt <= 10;                    // 跳过奇校验位
    end else begin                         // bit_cnt == 10：停止位
        byte_rdy <= 1; byte_val <= shift; bit_cnt <= 0;
    end
end
```

**过滤 break 码**：

```verilog
if (byte_rdy) begin
    if (byte_val == 8'hF0)
        got_f0 <= 1;                      // 记录：下一帧是松开码
    else if (got_f0)
        got_f0 <= 0;                      // 丢弃松开码（key release）
    else begin                            // 正常 make 码
        case (byte_val)
            8'h45: begin kbd_data<=0; kbd_valid<=1; end  // 数字 0
            8'h16: begin kbd_data<=1; kbd_valid<=1; end  // 数字 1
            ...
        endcase
    end
end
```

**Set 2 扫描码（数字键 0–9）**：

| 键 | 扫描码 | 键 | 扫描码 |
|---|---|---|---|
| 0 | `0x45` | 5 | `0x2E` |
| 1 | `0x16` | 6 | `0x36` |
| 2 | `0x1E` | 7 | `0x3D` |
| 3 | `0x26` | 8 | `0x3E` |
| 4 | `0x25` | 9 | `0x46` |

### 4.3 输出

| 信号 | 说明 |
|------|------|
| `kbd_data[3:0]` | 解码后的数字（0–9） |
| `kbd_valid` | 高有效，持续 **1 个时钟周期** |

`sales_v3` 和 `admin_mode` 共享这两个信号，各自在正确的状态下响应。

## 5. seg7_mux.v — 8 位数码管驱动

### 5.1 时分复用原理

EGO1 有 8 个数码管，但共享段线（segment lines），通过快速轮流点亮（时分复用）产生"同时显示"的视觉效果。同一时刻只有 1 位数码管真正点亮，其余位选关闭。

```
时钟  →  扫描计数器 cnt
         cnt 满了 → digit 递增（0→1→2→...→7→0→...）
         
digit 决定：
  - an[7:0]：哪一位位选高（= 1<<digit）
  - char_id：从 disp_data 中提取对应的字符 ID
  - seg0/seg1：字符 ID 转段码，送到对应段线
```

**扫描频率**：`SCAN_HZ=800`，即 1 秒内每个数码管被点亮 800/8=100 次（频率 100 Hz），远高于人眼感知下限（约 50 Hz），视觉上全亮。

### 5.2 字符 ID 编码

`disp_data[39:0]` 是 8 个 5-bit 字符 ID 的拼接，`[39:35]`=最左位，`[4:0]`=最右位。

| ID | 字符 | ID | 字符 | ID | 字符 |
|---|---|---|---|---|---|
| 0–9 | 数字 0–9 | 10 | A | 11 | b |
| 12 | C | 13 | d | 14 | E | 15 | F |
| 16 | H | 17 | L | 18 | O | 19 | P |
| 20 | S | 21 | t | 22 | r | 23 | 空白 |
| 24 | -（横线）| 25 | U | 26 | n | 27 | I |
| 28 | Y | | | | |

### 5.3 引脚布线细节（XDC 与数码管物理对应）

EGO1 数码管编号与 XDC 引脚的对应关系**是左右翻转**的：

- `an[7]`（软件最高位，最左字符）→ G2（物理最左位 BIT1）
- `an[0]`（软件最低位，最右字符）→ G6（物理最右位 BIT8）

同时，EGO1 数码管分为两组硬件段线：
- DN0：驱动左 4 位（BIT1–BIT4）→ 连接到 `seg1`（注意 seg1 驱动的是物理左半侧）
- DN1：驱动右 4 位（BIT5–BIT8）→ 连接到 `seg0`

代码中，`digit[2]==0` 时（digit 0–3，即右边 4 位）用 `seg0`，`digit[2]==1` 时（digit 4–7，即左边 4 位）用 `seg1`。

### 5.4 字符 ID → 段码转换

```verilog
function [7:0] id_to_seg;  // {dp, g, f, e, d, c, b, a}，高有效
    case (id)
        5'd0:  return 8'h3F;  // 0 → 0011_1111
        5'd1:  return 8'h06;  // 1 → 0000_0110
        5'd10: return 8'h77;  // A → 0111_0111
        5'd23: return 8'h00;  // blank → 全灭
        5'd24: return 8'h40;  // - → 只有 g 段
        ...
    endcase
endfunction
```

## 6. buzzer_driver.v — 报警音驱动

### 6.1 功能

`alarm_trigger=1` 时生成 **1320/1760/2200 Hz 三音递升循环报警音**；`aud_sd` 始终为高（使能板载音频放大器）。

### 6.2 PWM 方波生成原理

数字音频输出是一路方波，**频率决定音调**，占空比 50%。

```
目标频率 f → 半周期 = CLK_HZ / (2 × f) 个时钟周期
100 MHz → 1320 Hz 半周期 = 100,000,000 / 2640 ≈ 37,879 个周期

tone_cnt 从 0 计到 half_period-1，到达时：
  tone_cnt 清零
  aud_pwm 取反（产生方波）
```

```verilog
localparam HALF_0 = CLK_HZ / (2 * 1320);  // 37879 cycles → 1320 Hz
localparam HALF_1 = CLK_HZ / (2 * 1760);  // 28409 cycles → 1760 Hz
localparam HALF_2 = CLK_HZ / (2 * 2200);  // 22727 cycles → 2200 Hz
localparam TONE_DUR = CLK_HZ / 1000 * 150; // 每音持续 150 ms
```

### 6.3 三音轮换

```verilog
// dur_cnt 计满 TONE_DUR 后切换到下一音：0→1→2→0→...
if (dur_cnt >= TONE_DUR-1) begin
    dur_cnt <= 0;
    tone_sel <= (tone_sel == 2) ? 0 : tone_sel + 1;
end
```

### 6.4 报警停止

`alarm_trigger=0` 时，所有计数器清零，`aud_pwm=0`（静音）。下次触发从头开始。

## 7. sim_clk_wiz_0.v — MMCM 仿真桩

### 7.1 存在原因

Vivado 综合时使用真正的 `clk_wiz_0` IP（Clocking Wizard 生成的模块，内部例化 `MMCME2_ADV` 原语）。但 iverilog 等行为仿真器不认识这个 Xilinx 专有原语。

**解决方法**：提供一个行为模型，只要"输入 100 MHz → 输出 25 MHz"（精度不重要，仿真不测 VGA 像素时序）。

### 7.2 实现

```verilog
module clk_wiz_0 (
    input  wire clk_in1,   // 100 MHz
    input  wire reset,
    output wire locked,    // 仿真中永远是 1
    output wire clk_out1   // ≈25 MHz
);
    assign locked = 1'b1;
    reg [1:0] cnt = 0;
    always @(posedge clk_in1) cnt <= cnt + 1;
    assign clk_out1 = cnt[1];  // 100 MHz ÷ 4 = 25 MHz
endmodule
```

`cnt[1]` 每 2 个 100 MHz 周期翻转一次，产生 25 MHz 方波。

### 7.3 使用注意

- **仿真**：iverilog 编译时加入此文件
- **综合**：Vivado 中将此文件设为 "Simulation Only"，使用真正的 IP

## 9. vending.xdc — 约束文件

XDC（Xilinx Design Constraints）文件告诉 Vivado：
1. 哪个引脚连什么信号
2. 时钟频率是多少
3. 特殊时序要求（跨时钟域等）

### 9.1 配置电压

```tcl
set_property CFGBVS VCCO        [current_design]  # 配置 bank 由 VCCO 供电
set_property CONFIG_VOLTAGE 3.3 [current_design]  # 3.3V
```

EGO1 bank 0 供电为 3.3V，不声明会触发 `DRC CFGBVS-1` 警告。

### 9.2 时钟定义

```tcl
create_clock -period 10.000 -name sys_clk [get_ports clk]
# 告诉 Vivado：clk 引脚是 100 MHz 时钟（周期 10 ns）
# Vivado 据此计算所有寄存器之间的建立/保持时间要求
```

`clk_wiz_0` IP 内部会自动声明 25.175 MHz 的 `clk_out1_clk_wiz_0`，无需手动写。

### 9.3 跨时钟域约束

```tcl
set_clock_groups -asynchronous \
    -group [get_clocks sys_clk] \
    -group [get_clocks clk_out1_clk_wiz_0]
```

**为什么需要这条**：100 MHz 和 25.175 MHz 的比值是 3.97（非整数），没有公共时钟周期。Vivado 会对所有从 100 MHz 寄存器出发、到达 25.175 MHz 寄存器的路径，计算极端的 ~0.035 ns 建立时间要求（因为它只能找到一个非常小的"公共周期"），这个要求任何电路都不可能满足，导致大量误报的 Timing Violation。

`-asynchronous` 告知 Vivado 这两个时钟域间的路径不需要计时。**域内路径仍然正常约束**。

### 9.4 S6 按键特殊性

```tcl
set_property PACKAGE_PIN P15 [get_ports {btn[5]}]
# P15 是 FPGA_RESET 专用引脚，板上有 4.7kΩ 上拉
# 未按时 P15=高，按下时 P15=低（接地）
# 顶层：btn_debounce u_deb5(.btn_in(~btn[5]), ...)  ← 取反处理
```

### 9.5 数码管引脚交叉

数码管引脚分配故意与 EGO1 默认编号**相反**，使得 `an[7]`（顶层最高位，对应最左字符）物理上对应最左侧数码管（BIT1/G2）。这样数码管从软件视角和物理视角都是"左边是高位"，符合直觉。

```tcl
set_property PACKAGE_PIN G2 [get_ports {an[7]}]  # BIT1，物理最左
set_property PACKAGE_PIN G6 [get_ports {an[0]}]  # BIT8，物理最右
```

同时 `seg0` 接 DN1 段线（驱动物理右侧），`seg1` 接 DN0 段线（驱动物理左侧），与 `seg7_mux` 中 digit 0–3 用 seg0、digit 4–7 用 seg1 的逻辑配套。
